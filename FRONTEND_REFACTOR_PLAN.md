# خطة إعادة تنظيم الواجهة

## نتيجة الفحص الحالي

المستودع الحالي ليس مشروع Django SSR حسب الفحص المحلي الحالي. البحث في المجلد الأب القريب لم يجد مشروع Django داخل مجلد عسير، والبحث الأوسع في سطح المكتب وجد مشروع Django خارجي باسم `spoort` يبدو غير مرتبط بمنصة عسير الصحية.

- لا يوجد ملف `manage.py`.
- لا توجد مجلدات Django المعتادة مثل `templates/` أو `static/` في جذر المشروع.
- لا توجد مؤشرات Django مثل `INSTALLED_APPS` أو `STATIC_URL` أو `csrf_token` أو `urlpatterns`.
- `package.json` يوضح أن المشروع الحالي React + Vite + TypeScript، مع Express في مجلد `server/`.
- نقطة تشغيل الواجهة هي `src/main.tsx` وتستخدم React Router في `src/routes/AppRoutes.tsx`.
- ملف CSS الرئيسي الحالي هو `src/index.css`.
- مشروع Django الخارجي الموجود في `C:\Users\Test2\OneDrive\Desktop\spoort` لم يتم تعديله لأنه خارج مسار العمل الحالي ولا يظهر مطابقا لمشروع عسير.

## قرار أمان قبل التعديل

لن أنشئ بنية Django مثل `templates/base.html` أو `static/css/*.css` داخل هذا المستودع إلا إذا تأكدنا أن هذا هو مشروع Django المقصود أو تم توفير مساره الصحيح. إنشاء هذه البنية الآن سيكون تغييرا معماريا غير مستخدم من التطبيق الحالي وقد يربك النشر الحالي على GitHub Pages/Vite.

كذلك لن أنفذ المرحلة الكبيرة الخاصة بقاعدة البيانات والصلاحيات ولوحة التحكم الكاملة قبل تثبيت المسار المعماري، لأن `123/yaser.md` يطلب منصة كاملة بقاعدة بيانات وAPI ولوحة إدارة، بينما هدف Django SSR يفترض بنية مختلفة تماما.

## إذا كان هذا هو المشروع المقصود فعلا

الخطة الآمنة تكون بصيغة مناسبة لـ React/Vite مع الحفاظ على التصميم الحالي:

1. الهوية البصرية:
   - إنشاء ملفات منظمة تحت `src/styles/` مثل:
     - `tokens.css`
     - `base.css`
     - `components.css`
     - `pages.css`
   - جعل `src/index.css` يستورد هذه الملفات فقط أو يبقى كملف تجميع.
   - توحيد ألوان عسير الرسمية من `123/yaser.md` والشعارات والظلال والمسافات بدون تغيير تدفق الصفحات.
   - استبدال الاستخدامات الزائدة لـ `green/emerald/teal/amber/orange` بدرجات الهوية الرسمية، مع إبقاء ألوان التنبيه فقط عند الحاجة.

2. المكونات المتكررة:
   - استخراج عناصر متكررة من `ClientLayout` و `AdminLayout`.
   - توحيد أزرار الاختيار، البطاقات، الترويسات، شريط التنقل السفلي، وحاويات الصفحات.

3. الصفحات الكبيرة:
   - تقسيم `AdminDashboardPage.tsx` إلى أقسام أصغر داخل `src/components/admin/`.
   - تقسيم `QrLocationsAdminPage.tsx` إلى نموذج، جدول/قائمة، وحالات تحميل/خطأ.
   - عدم تغيير المسارات الحالية في `src/routes/AppRoutes.tsx`.

4. الحالة والبيانات:
   - تقسيم `src/store/appStore.ts` بحذر إلى slices أو helpers إذا كان ذلك لا يغير API المستخدمة في الصفحات.
   - تنظيم `mockData.ts` حسب المجال بدون تغيير أسماء الصادرات المستخدمة.

5. JavaScript والأمان:
   - إبقاء منطق الواجهة داخل ملفات TypeScript الحالية.
   - مراجعة أي استخدامات خطرة مثل `dangerouslySetInnerHTML` إن وجدت.
   - عدم إدخال سكربتات inline جديدة.
   - إزالة كلمات المرور والأسرار الافتراضية من `server/auth.js` وجعلها إلزامية عبر متغيرات البيئة قبل الإنتاج.
   - مراجعة fallback credentials في الواجهة والتأكد أنها لا تعمل في الإنتاج.

6. التحقق بعد كل مرحلة:
   - تشغيل `npm run build`.
   - فتح الصفحات الأساسية: الرئيسية، رحلتي، الجواز، الفعاليات، المساعد، ولوحات الإدارة.
   - فحص الجوال خصوصا ظهور الشعارات والألوان وعدم كسر التخطيط.

## إذا كان المطلوب مشروع Django SSR آخر

عند توفير مسار مشروع Django الصحيح، تكون الخطة:

1. فحص `manage.py` وملفات `settings.py` و`urls.py` والتطبيقات.
2. جرد القوالب الحالية ومسارات `static`.
3. إنشاء/تنظيم:
   - `templates/base.html`
   - `templates/includes/`
   - `templates/components/`
   - `templates/pages/`
   - `static/css/tokens.css`
   - `static/css/base.css`
   - `static/css/components.css`
   - `static/css/pages.css`
   - `static/js/main.js`
4. نقل CSS/JS المكرر تدريجيا مع الحفاظ على RTL وCSRF والنماذج والمسارات.
5. منع `|safe` غير الضروري والسكربتات الخطرة.
6. التحقق بعد كل مجموعة عبر:
   - `python manage.py check`
   - تحميل الصفحات المهمة
   - اختبار النماذج وCSRF
   - فحص static files

## نطاق المرحلة الحالية

تم تنفيذ الفحص فقط وتوثيق الخطة. لم يتم تنفيذ التعديلات الكبيرة بعد بانتظار تأكيد المسار الصحيح أو اعتماد خطة React/Vite البديلة لهذا المستودع.

## ملاحظة اختبار

لا يمكن تشغيل `python manage.py check` داخل هذا المسار لأن `manage.py` غير موجود. إذا تم اعتماد هذا المستودع الحالي، يكون اختبار المرحلة عبر `npm run build` وفحص الصفحات. إذا تم توفير مشروع Django SSR الصحيح، يعود اختبار المرحلة إلى `python manage.py check` كما هو مطلوب.
